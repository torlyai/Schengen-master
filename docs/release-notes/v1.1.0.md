# Visa Master v1.1.0 — Premium: Auto-Book Tier

> **Release type:** Feature release (introduces the paid Premium tier alongside the unchanged Free tier)
> **Date:** 2026-05-14 (Chrome Web Store submission)
> **Tag:** `v1.1.0`
> **Free tier:** Identical core behaviour to v1.0.9, plus three new free quality-of-life features (Cloudflare detection, eight new optional Telegram pings, BYO webhook channel). No forced migration. No new permissions if you don't activate Premium.
> **Premium tier:** Opt-in. £0 install. £19 charged **only on a successful auto-booking**.
> **Languages:** Popup + Welcome page fully translated (English / 简体中文).
> **Brand:** New ApertureMark glyph replaces the typographic "v" — visible on the toolbar, popup, settings, welcome page, and Premium intro tab.

---

## TL;DR

After nine patch releases hardening the free notifier, Visa Master now has a second tier. **Premium auto-books the slot for you** — no clicking through the TLScontact booking flow at 3am, no missed appointments because you blinked. The auto-book transaction still runs **inside your browser, in your authenticated session**, on the same MIT-licensed codebase. The only thing on our server is a tiny licensing API that issues a signed token after Stripe payment.

If you only want the free slot-notifier you already love — **do nothing**. Your install upgrades to v1.1.0 with zero behavioural change.

If you want the slot booked automatically the second it opens — open the popup, click "Tell me more about Premium," and follow the four-step setup.

## Building Visa Master — from zero to now

This is the project's first major release. Here's the full arc, in prose, so anyone landing on this page understands not just *what* shipped but *why* it took the shape it did.

### Chapter 1 — The itch (pre-v1.0.0, late 2025)

Booking a Schengen visa appointment through TLScontact is a refresh-the-page game. Slots open at unpredictable times, fill within seconds, and the official UI gives you nothing — no waitlist, no notification, no "open soon" indicator. The existing commercial extensions (TLSContact Appointment Booker; later VisaReady) solve this by routing your TLS session through a server-pool that does the watching and booking on your behalf — which works, but requires giving an unknown vendor your TLS password and a session on their backend.

The premise of Visa Master was simple: **the watch loop is a Chrome extension's job, not a server's job.** A service worker, a single content script, `chrome.alarms`. No backend. No credentials. Notify the user the second a slot appears, let them book it themselves.

### Chapter 2 — The first release (v1.0.0, `cb3c2e9`)

Initial commit shipped the watcher: service worker brain, content-script slot detector reading the TLScontact DOM, popup React UI in five states (`UNKNOWN` / `LOGGED_OUT` / `NO_SLOTS` / `SLOT_AVAILABLE` / `PAUSED`), desktop notifications via `chrome.notifications`. MIT-licensed from day one — the trust posture (auditable, local-only) was the product.

Two structural decisions made in v1.0.0 turned out to be load-bearing for everything that came after:

1. **A single canonical state enum in `shared/states.ts`.** Both the service worker and the popup read from it. This made it trivial later to add `WRONG_PAGE` and twelve Premium states without breaking the existing rendering pipeline — every new state was a new switch case, never a refactor.
2. **`chrome.storage.local` as the only source of truth.** No in-memory caches, no Redux. Every state mutation goes through a typed wrapper in `shared/storage.ts`. When v1.1.0 needed to add `bookingWindow` and `licenseToken` keys, it was a 30-line addition with zero risk to existing keys.

### Chapter 3 — Phones, not just laptops (`183a132`, v1.0.4)

The first piece of post-v1.0.0 feedback was unanimous: "desktop notifications don't reach me when I'm on the tube." `183a132` added the **Telegram bot integration** — a long-lived bot token in settings, a chat ID, and the service worker now pushes alerts to your phone the same instant it surfaces them on desktop. v1.0.4 (`869690d`) polished this further: pause notifications, local-time timestamps, so the message you wake up to actually tells you when the slot opened in your timezone.

This is the moment Visa Master went from "a tab thing" to "a thing that reaches you anywhere."

### Chapter 4 — Internationalisation by default (`ad153b8`, v1.0.1 polish)

The user base that needs Schengen visas the most is overwhelmingly non-English-native. Rather than retro-fit i18n later, the install/config guide was made bilingual (EN + 简体中文) at v1.0.1, and the popup string table was structured for translation from then on. Every state label, every notification body, every welcome-page paragraph was added in both languages in the same commit. v1.1.0 inherits this discipline — the +327 added strings on each side of `en.json` / `zh.json` arrive in lockstep.

### Chapter 5 — Scope discipline (v1.0.2, `efb6e38`)

The hardest commit in the history. v1.0.1 had bolted on an OpenClaw integration — a then-promising path to extending the watcher with more advanced detection. v1.0.2 ripped it back out. The reasoning in the commit message: *"v1 simplification."* Fewer dependencies meant a smaller bundle, easier audit, fewer permission requests, and — crucially — fewer surfaces where a future Premium tier could leak. Hindsight says this was the right call: the v1.1.0 Premium tier wouldn't have been auditable if v1.0.x had grown a second runtime alongside it.

### Chapter 6 — The polish run (v1.0.3 → v1.0.7, six commits)

Four months of weekly patch releases hammering on the small things that decide whether a free tool stays installed:

- `656c8e9` (v1.0.3) — pause / resume / acknowledge buttons actually re-rendered the popup. Before this, they fired the right message but the UI didn't reflect the response. Looked broken; wasn't, but might as well have been.
- `565977f` — the contact-QR images for Telegram setup got a hover affordance and a bottom-of-popup Settings shortcut. Discoverability fix.
- `8660397` (v1.0.5) — those same QR images weren't in the release bundle. CI fix. Embarrassing, fixed, moved on.
- `1172a14` (v1.0.6) — the popup's bottom chrome (Settings link, branding, QR row) wasn't pinning correctly at narrow widths. Flex-pinning, smaller QRs, brand link to torly.ai. The popup now renders correctly from 280px to 480px wide.
- `469fee5` (v1.0.7) — the "no slots" status indicator was a filled `●`, which read as an alarm when it should have read as "all calm." Swapped for `•`. One character, materially better signal.

None of these are flashy. All of them are why the rating curve doesn't drop in month two.

### Chapter 7 — Self-awareness (v1.0.8, `37f57e1`)

By v1.0.8 there were users running v1.0.3 alongside users running v1.0.7. A side-loaded extension has no auto-update path, and asking users to manually check GitHub for releases is asking too much. v1.0.8 added a **Check for updates** button on the Settings page that hits the GitHub Releases API, compares the tag against `manifest.json`, and surfaces an inline pill if a newer version exists. This is also the foundation v1.1.0 leans on to ask Premium users to upgrade when the auto-book FSM gets meaningful improvements.

### Chapter 8 — The reliability fix (v1.0.9, `e0cbf5d`)

The quietest commit in the history and possibly the most important. Chrome aggressively evicts service workers on idle laptops; on macOS, the OS layer adds App Nap on top. The combined effect: Visa Master's poll alarm could die overnight without surfacing any error. The watcher would *look* alive — popup rendering fine, last-check timestamp visible — but no actual TLS checks were happening.

v1.0.9 added a self-healing alarm scheduler that detects when the next-check timestamp has slipped past `Date.now() + maxJitter`, re-arms the alarm, and emits a recovery log line. It also stopped racing `chrome.alarms` — earlier code was firing duplicate alarms when the SW restarted, which Chrome silently de-duplicated, which masked the underlying problem. The fix is invisible to users and load-bearing for everything Premium adds in v1.1.0 — auto-booking requires the watcher to *actually* be running.

### Chapter 9 — Why Premium, why now (v1.1.0, this release)

By April 2026 the free tier was stable, multilingual, multi-channel, and self-healing. The Schengen-booking landscape, meanwhile, was consolidating: VisaReady went live 2026-05-13 with a server-coordinated £29 auto-book model. TLSContact Appointment Booker had grown to 6,000 users at 3.2★ — clear demand, clear dissatisfaction.

The architectural question Premium had to answer: **can we offer auto-booking without abandoning the local-only promise that defines the free tier?**

The answer, in v1.1.0:

1. **Scanning stays local.** Same content script, same DOM detector, same `chrome.alarms` scheduler. Free and Premium share the watch path byte-for-byte.
2. **The booking transaction stays local.** When a matching slot is detected, the booking FSM drives the TLScontact page in the user's own authenticated tab. We never log in to TLS from our servers.
3. **The only server is a billing API.** A small Next.js app at `api.torly.ai` issues a signed JWT licence after Stripe payment, accepts a `bookingId` for capture (£19 charge) and refund, and stores nothing about the user beyond what Stripe billing requires.
4. **Free users see no behavioural change.** No new permissions, no new code paths exercised, no migration. Their install upgrades to v1.1.0 and continues to behave exactly as v1.0.9 did.

The trade-off — and the one place Premium genuinely diverges from the free tier's promise — is **credential storage.** Premium needs your TLS email + password to re-log you in when TLS expires your session (which it will, repeatedly). Those credentials are AES-encrypted with a per-install salt and live only in `chrome.storage.local`. They never leave the browser. This is restated explicitly in the activation flow, the welcome page, and `docs/09-visa-master-premium-prd.md` §11. Users who don't want any credential storage stay on Free — and nothing about the free experience changes.

This is the shape of the product as of 2026-05-13. Free forever. Premium when you need it. Local-first, always.

## Why this release matters

Two things happened in the Schengen-booking landscape this quarter that forced the question:

1. **VisaReady launched 2026-05-13** with a £29 success-fee auto-book model, charging the booking flow through their own server pool. 26 users on day one.
2. **TLSContact Appointment Booker has 6,000 users at 3.2★** — clear demand for auto-booking, equally clear dissatisfaction with how it's currently delivered.

Both incumbents are thin clients that depend on a coordinated server pool — they need your TLScontact credentials and a session on their backend to do their work. We had a choice: stay free-only and watch the auto-book segment consolidate around closed-source, server-coordinated tools — or ship a Premium tier that does auto-booking *without* abandoning the local-only trust posture that 1.0.x users installed Visa Master for.

This release is that second answer.

See [`docs/08-vs-alternatives.md`](../08-vs-alternatives.md) for the head-to-head comparison and [`docs/09-visa-master-premium-prd.md`](../09-visa-master-premium-prd.md) for the full PRD.

## What's new in v1.1.0

### Premium — Auto-Book Tier *(new, opt-in)*

When a slot matching your travel window opens, Premium drives the TLScontact booking flow inside your existing browser tab and confirms the appointment. You're notified within seconds on desktop + Telegram. You then have ~30 minutes to pay the TLS service + consulate fee on TLS's own site.

**Pricing.** £0 install. £19 charged only when we successfully book a slot. Refund window: 24h, in-popup, no email required. £0 if no slot was bookable.

**Activation flow** *(4 steps, ~90 seconds end-to-end)*:

1. **Preflight** — Confirms you're on TLScontact and the watcher is healthy.
2. **Credentials** — Enter your TLScontact email + password. Stored AES-encrypted in `chrome.storage.local`, never sent to our servers. (This is the one place Premium diverges from the Free tier's "we never touch credentials" promise — see *Privacy* below for how we contain it.)
3. **Sign-in test** — Premium opens your TLS tab and verifies the credentials work. If TLS expires your session later, Premium re-logs in automatically.
4. **Booking window** — Travel date + processing-day buffer + min-notice. We only auto-book slots that fall inside this window.

After step 4, Stripe Checkout opens in a new tab. On payment success we redirect you to `torly.ai/visa-master/activated`, a tiny page whose only job is to relay a signed JWT licence token back into the extension. From that moment your popup shows `Active` instead of `No slots`, and the watcher will book on your behalf when criteria are met.

**Booking states added to the popup** (PRD `docs/09` §12, wireframes `docs/10`):

```
PREMIUM_PREFLIGHT          PREMIUM_OPTIONS
PREMIUM_SETUP_CREDENTIALS  PREMIUM_BOOKING_IN_PROGRESS  (60s budget)
PREMIUM_SETUP_SIGNING_IN   PREMIUM_BOOKED               (24h refund window)
PREMIUM_SETUP_BOOKING_WINDOW
PREMIUM_SETUP_READY        PREMIUM_BOOKING_FAILED       (no charge)
PREMIUM_VERIFICATION_GATE  PREMIUM_REFUND_PROMPT
PREMIUM_SETUP_FAILED_RETRY
PREMIUM_SETUP_FAILED_STALE PREMIUM_ACTIVE
```

**Invariants enforced by the booking FSM** (`extension/src/background/booking-fsm.ts`):

- 60-second total budget per auto-book attempt — if the DOM doesn't behave, we abort and leave you to book manually.
- One booking per 24h per install. Cooldown is enforced after a successful capture so a Stripe webhook hiccup can't double-charge you.
- Licence-gated at both the FSM entry **and** the `/booking/capture` call. Two independent gates so neither alone is load-bearing.
- Idempotent on `bookingId`. The Vercel backend's database is the source of truth — re-tries are safe.

### Free tier — also new in v1.1.0

- **WRONG_PAGE detection.** Previously, if you logged in to TLScontact but ended up on `/travel-groups`, `/applications`, or `/profile`, the popup classified the page as `UNKNOWN` and showed a useless "is this the right page?" prompt. v1.1.0 detects the sub-page family explicitly and offers a one-click CTA to navigate to the booking page.
- **Cloudflare challenge detection.** TLScontact intermittently puts the page behind a Cloudflare challenge ("Verifying you are human..."). v1.1.0 surfaces this as a first-class state (`CLOUDFLARE`) with a desktop notification, a Telegram alert (opt-in), and a 15-minute watchdog that falls back to IDLE if you don't resolve the check.
- **Eight new optional Telegram pings.** Beyond the existing slot-found + blocker pings, you can now opt into Telegram for: monitoring started/resumed/paused, page-needs-classification (`UNKNOWN`), wrong-page (`WRONG_PAGE`), auto-stop after blocker timeout, and (Premium only) booking-in-progress, refund prompt, license events, auto-login pause. All gated behind per-event toggles with sensible defaults (auto-stop = on; monitoring lifecycle = off). See `docs/14-notifications-and-email-prd.md` §6 for the full matrix.
- **BYO Webhook channel.** Free users can now opt to POST every event to a user-supplied HTTPS URL (Slack, Discord, IFTTT, Pipedream, your own server). Optional HMAC-SHA256 signing via a shared secret. **Preserves the "Free never calls torly.ai" promise** — the URL is yours, not ours. Data minimisation: never includes TLS credentials, polling cadence, or DOM contents.
- **Brand refresh.** The "v" wordmark glyph in toolbar, settings, welcome and Premium intro pages is replaced by the new **ApertureMark** (outer ring + green wedge + center dot). Matches the new icon set (16/32/48/128 px PNG, available in the release ZIP).
- **Premium upsell line on `SLOT_AVAILABLE`.** A single quiet line above the footer offering "Want this booked automatically next time?" Free users can ignore it and never see it again after dismissing.
- **Bilingual welcome page.** The post-install welcome page now describes both tiers (Free vs Premium), with EN and 简体中文 strings shipped at parity.
- **Popup ⋯ More button.** Renamed and re-purposed — for Free users it opens the existing full-tab Settings page (unchanged). For Premium users it opens the in-popup Options tab (`PREMIUM_OPTIONS`). Two surfaces, one button, context-aware.
- **More CSS.** ~1,450 lines added covering the Premium popup surfaces (upsell card, trust callouts, step progress, hero banners, refund radio list). All scoped under Premium-specific class names — Free-tier styling unchanged.
- **Version display correctness.** The Settings page was hardcoded to show "v1.0.0" regardless of the actual installed version. Now reads from `chrome.runtime.getManifest().version`, so "Check for updates" actually compares meaningful numbers.

## Architecture & trust posture

Premium uses a **hybrid** architecture, NOT a server-coordinated network:

```
   Your laptop                           torly.ai (Vercel)
  ┌──────────────────┐                 ┌────────────────────┐
  │ Visa Master      │  HTTPS, only    │ /api/visa-master/  │
  │ extension        │ ──────────────► │  checkout          │
  │  • scans TLS     │  on Premium     │  license/*         │
  │  • books TLS     │  events:        │  booking/*         │
  │  • holds creds   │                 │  webhook/stripe    │
  └──────────────────┘  • activate     └────────────────────┘
   (still 100% local   • capture £19         │
    for Free users)    • refund              ▼
                                       Supabase Postgres
                                       (licences, bookings)
                                       Stripe (payments only)
```

- **No shared scanning network.** Each Premium install scans alone. No coordinated polling pool.
- **No server-side scrape of TLScontact.** We never log in to TLS from our servers. The TLS session lives only in your browser tab.
- **No user content on our servers** beyond what Stripe captures for billing (email, card metadata) and a `bookingId` for idempotency.
- **Credentials stored only locally**, AES-encrypted with a key derived from a per-install salt.

Full architectural detail in [`docs/11-architecture.md`](../11-architecture.md).

## Changed permissions

If you're on Free tier, you'll see **no new permissions prompt**. The Free codepath uses the same `*.tlscontact.com` host permission as v1.0.9.

If you activate Premium, the extension requests:

- `host_permissions: https://torly.ai/*, https://*.torly.ai/*` — for licence activation and the £19 booking-capture API call.
- A content script on `https://torly.ai/visa-master/activated*` — the one-page relay that injects the signed JWT licence token back into the extension after Stripe Checkout. It reads nothing else.

These are added to the manifest in v1.1.0 but only exercised on Premium activation.

## Breaking changes

**None for Free users.** v1.0.9 storage shape, state names, and message types are preserved unchanged. Stats counters, consent records, and target storage all continue to read clean.

The single new top-level storage key is `bookingWindow` — Premium-only, ignored by the Free tier code path.

## Known limitations

- **One TLS account per install.** Family of three → three browser profiles, three installs. Multi-applicant booking is explicitly out of scope for v1.1 (see PRD §3.2).
- **Auto-fill stays manual.** Premium books the appointment slot only; passport details, photos, and the application form itself remain your responsibility.
- **TLScontact only.** No VFS Global, no BLS. France / Germany / Belgium / Netherlands / Spain / Italy via TLS.
- **Desktop Chrome only.** Telegram bot remains the phone-notification path.
- **TLS login selectors are best-guess** and need real-world validation. The auto-login module fails fast if it can't find the expected email / password / submit elements — better to ask you to log in manually than to click the wrong button.

## Installation

**Fresh install:**
1. Download the release bundle (`visa-master-1.1.0.zip`) from the assets below.
2. Unzip to a folder you'll keep around (Chrome reads from disk every reload).
3. Visit `chrome://extensions`, enable **Developer mode**, click **Load unpacked**, point at the unzipped folder.
4. The welcome tab will open. Pick your language and a TLS URL to watch.

**Upgrading from v1.0.x:**
1. Download `visa-master-1.1.0.zip`.
2. Unzip over your existing folder, or to a new folder and re-point `chrome://extensions`.
3. Click the reload button on the extension card.
4. Your settings, paused state, and Telegram config persist.

**Web Store** *(pending listing update)*: search for "Visa Master — Appointment Watcher" on the Chrome Web Store. Free tier identical to side-loaded build; Premium activation works the same way.

## What's bundled into the v1.1.0 submission (delta vs the original Premium-tier draft)

The original v1.1.0 release notes were drafted on 2026-05-13 when the Premium FSM first landed. Between that draft and the actual Chrome Web Store submission on 2026-05-14, the following work was folded into the same v1.1.0 ZIP rather than being held back for a v1.1.1 patch:

| Area | Commit | Why it shipped with v1.1.0 |
|---|---|---|
| Wizard fixes — Start-setup CTAs + wizard advance | `380b1ab`, `dadb513` | Premium activation was broken in the original draft; can't ship a broken Premium tier. |
| Premium UX polish, welcome redesign, scoped CSS | `0dd6ad8` | The original draft had functional but unbranded Premium surfaces. Listing screenshots needed the polished look. |
| ApertureMark extraction + brand-mark swap | `bf4e7ec` | The Chrome Web Store listing screenshots are the user's first impression. Shipping the typographic "v" alongside the new icon set in `/icons/` was inconsistent. |
| Cloudflare challenge detection | `a53d013` | TLScontact's Cloudflare gate has been hitting Free users regularly. Detecting it cleanly was the difference between "broken extension" complaints and a managed UX. |
| Notifications & Email PRD + extension hooks | `3e41fba`, `1466761`, `a296925` | The PRD locks decisions for v1.2.0 backend work. The two extension-side phases (N-1 Telegram gap-fill, N-6 BYO webhook) are user-facing and zero-cost to ship now — they activate immediately even without backend deployed. |
| Version-display fix on Settings page | `88cc9f6` | The screenshot run surfaced a hardcoded "v1.0.0" string. Caught during listing prep, fixed in the same window. |
| Playwright harness for listing screenshots | `c9ccdd4`, `23f497f`, `88cc9f6` | Producing reliable, branded screenshots for the Web Store form. Side benefit: the harness is the start of our test infrastructure for future releases. |
| Premium PRD / wireframes / architecture docs | `a1fdd3c` | Reference material the Premium tier needed before it could be reviewed and reasoned about by anyone except the original author. |

None of these were strictly required to ship the Premium tier, but the consolidated submission improves the user-visible polish and includes infrastructure (Telegram gap-fill, BYO webhook, brand mark) that becomes load-bearing for v1.2.0 (email notifications). Holding them for a v1.1.1 patch would have meant re-submitting to the Web Store twice in a week.
