# Visa Master — Appointment Watcher

Privacy-first Chrome MV3 extension that watches your TLScontact appointment
page and notifies you the instant a slot opens. The extension runs entirely
in your own browser — it never touches your credentials, never books for you,
and never sends raw page content off your device.

See `../docs/06-visa-master-chrome-extension-prd.md` for the full PRD and
`../docs/07-chrome-extension-wireframes.md` for screen-by-screen wireframes.

## Install (end users)

**Recommended:** download the pre-built ZIP — no Node, no build step.

1. Open the [latest release](https://github.com/torlyai/Schengen-master/releases/latest)
   and download `visa-master-v<version>.zip`.
2. Unzip it. You get one folder named `visa-master-v<version>/`.
3. Open `chrome://extensions` in Chrome.
4. Toggle **Developer mode** (top-right).
5. Click **Load unpacked** and pick the unzipped folder.
6. Open your TLS booking page (e.g.
   `https://visas-fr.tlscontact.com/workflow/...`) while logged in. The
   badge turns green, monitoring is on.

The Chrome Web Store listing is not live yet — until it is, "Load unpacked"
is the only install path Chrome allows.

Full bilingual end-user instructions: see the
[main README](../README.md#path-b--install-the-pre-built-extension-recommended)
(EN) or [中文 README](../README.zh-CN.md#安装步骤推荐路径--下载-zip) (中文).

## Develop

```sh
npm install
npm run build          # one-shot build into dist/
npm run package        # zip dist/ → visa-master-v<version>.zip
npm run release        # build && package — same as the GH Actions release flow
npm run typecheck      # tsc --noEmit, no emit
npm run dev            # vite dev with HMR (use chrome://extensions reload)
npm run icons          # regenerate placeholder icons
```

The ZIP packaging is a pure-Node script with no native deps — see
[`scripts/package.mjs`](scripts/package.mjs). It writes the ZIP next to
`package.json` (gitignored). For tagged releases the same script runs in
`.github/workflows/release.yml` and attaches the ZIP to the GitHub Release.

Build target: Chromium ≥ 110. Uses `@crxjs/vite-plugin` so the `manifest.json`
is the single source of truth for entry points (service worker, content
script, popup, welcome, settings).

## Architecture

```
content-script  →  service-worker  →  badge / notifications / OpenClaw
   (detector)       (state machine,
                     scheduler, SW
                     message router)
```

- **Content script** (`src/content/`) — runs on `*.tlscontact.com` pages,
  evaluates a multi-signal detector on every page load and DOM mutation, and
  sends `DETECTION_RESULT` to the service worker. Also receives commands to
  swap the tab title / favicon when a slot is found.
- **Service worker** (`src/background/`) — owns the state machine, the
  `chrome.alarms` polling schedule, the badge, desktop notifications, and the
  OpenClaw WebSocket client. State persists in `chrome.storage.local` and is
  restored on SW eviction.
- **Shared** (`src/shared/`) — the typed message contract that the popup and
  settings UI use to talk to the SW. UI agent imports types from here.

### Shared message contract (UI ↔ SW)

```
┌────────────┐                 ┌─────────────────┐                ┌────────────┐
│  Popup /   │  chrome.runtime │  Service        │  chrome.tabs   │  TLS tab   │
│  Settings  │   sendMessage   │  Worker         │  + scripting   │  (content  │
│  (UI agent)│ ─────Msg──────► │  (this agent)   │ ─────────────► │   script)  │
│            │ ◄────STATUS──── │                 │ ◄DETECTION_RES─│            │
└────────────┘                 └─────────────────┘                └────────────┘
```

All message types live in `src/shared/messages.ts`. The full discriminated
union is `Msg`; the status payload is `StatusPayload`; the settings payload
is `SettingsPayload`.

### Detection rule (TUNABLE)

The detection logic lives in `src/content/detector.ts`. Search the file for
`TUNABLE:` to find the classification rule — it's the block of code that
decides between `SLOT_AVAILABLE`, `NO_SLOTS`, and `UNKNOWN` based on how many
positive signals fired.

The rule is intentionally simple:

- Two or more positive signals out of `{bookEnabled, slotCount>0, noSlotsTextAbsent}`
  → `SLOT_AVAILABLE`.
- Zero positives + the no-slots phrase present → `NO_SLOTS`.
- Otherwise → `UNKNOWN` (the user is asked to classify it once via the popup).

`CLOUDFLARE` and `LOGGED_OUT` short-circuit everything else.

### Polling cadence

Lives in `src/background/scheduler.ts`. The `smart` mode uses the user's
configured release windows in UK local time (default 06:00–09:30 and
23:30–00:30) — inside a window we poll every 2 minutes, outside every 6.
`chrome.alarms` ticks fire `pollOnce()` which reloads the watched tab with
`chrome.tabs.reload()` — this preserves cookies and Cloudflare clearance
(empirically confirmed 2026-05-12, see PRD Appendix C).

## Icons

The icons in `public/icons/` are **placeholders** generated by
`scripts/generate-icons.mjs` — a black tile with a stylised "v". They are
intentionally crude. Replace them with the final brand assets before any
public release. The script has no native dependencies, so it works on every
platform without compiling `sharp`.

## What this directory owns

This sub-repo is split across two agents working in parallel:

- **This agent (backend):** `manifest.json`, `vite.config.ts`,
  `src/shared/**`, `src/background/**`, `src/content/**`, `public/icons/**`,
  build tooling.
- **UI agent:** `src/popup/**`, `src/welcome/**`, `src/settings/**`,
  `src/components/**`, `src/styles/**`, `src/i18n/**`.

Stub HTML/TSX files exist in the UI directories so `vite build` works; they
say `STUB — overwritten by UI agent` at the top. Once the UI agent fills
them in, those stubs vanish.
